# Immutable evidence artifact manifests

`tools/evidence/build_evidence_manifest.py` creates and verifies one offline,
regenerate-only manifest for a captured Shooter Mover evidence package. The
manifest binds the accepted EH-001 identity, EH-002 configuration and seed,
session/attempt identity, CS-012 validity and diagnostics bounds, EH-007
performance summary, the UF-010 Windows build, and every package file by
SHA-256.

The tool performs no network access, signing, upload, credential lookup,
release-store packaging, gameplay work, or Unity invocation.

## Package layout

The package root is caller-owned local output. The following names are reserved
at its root and are regenerated by this tool:

```text
evidence-manifest-v1.json
evidence-manifest-v1.sha256
```

All other regular files beneath the package root are inventory inputs. The
generator rejects symlinks/reparse-like paths, traversal, absolute paths,
backslashes, control characters, non-NFC path text, duplicate paths, and
case-only path collisions. Inventory paths always use `/` and UTF-8 byte-order
sorting.

A typical package contains:

```text
evidence-package.json
identity/evidence-identity.txt
configuration/stage1.json
diagnostics/summary.json
performance/summary.json
windows-build/
  ShooterMover.exe
  ShooterMover_Data/boot.config
  ShooterMover_Data/globalgamemanagers
  UnityPlayer.dll
  unity-build.log
  build-fingerprints.json
  build-artifacts.txt
```

The manifest and checksum are deliberately excluded from their own inventory.
Any other file added after generation is an extra file and verification fails.

## Descriptor v1

`--descriptor` is a package-root-relative path to a strict JSON object. Unknown
or duplicate fields are rejected.

```json
{
  "schema": "shooter-mover.evidence-package-descriptor",
  "version": 1,
  "identityRecordPath": "identity/evidence-identity.txt",
  "configurationPath": "configuration/stage1.json",
  "session": {
    "sessionId": "session-stage1-001",
    "attemptId": "attempt-001",
    "parentSessionId": null,
    "state": "Ended",
    "validity": {
      "status": "valid",
      "reasonCodes": []
    }
  },
  "diagnostics": {
    "summaryPath": "diagnostics/summary.json",
    "eventCount": 12,
    "maximumEventPayloadBytes": 512,
    "logBytes": 4096,
    "retainedLogCount": 1,
    "truncated": false
  },
  "performance": {
    "summaryPath": "performance/summary.json",
    "state": "Completed",
    "warmUpSeconds": 2.0,
    "captureSeconds": 10.0,
    "frameSampleCount": 600,
    "qualityProfile": "Medium"
  },
  "build": {
    "rootPath": "windows-build",
    "status": "succeeded",
    "complete": true,
    "exitCode": 0,
    "fingerprintsPath": "windows-build/build-fingerprints.json",
    "artifactInventoryPath": "windows-build/build-artifacts.txt",
    "executablePath": "windows-build/ShooterMover.exe"
  }
}
```

Validity reason codes use CS-012-style stable IDs and must be sorted and unique.
A technically invalid ended session may still be manifested for review, but its
top-level `artifactStatus` is `invalid`; `--require-valid` rejects it.

## Build

From the repository root:

```powershell
python .\tools\evidence\build_evidence_manifest.py build `
  --package-root "C:\evidence\stage1-run-001" `
  --descriptor "evidence-package.json" `
  --require-valid
```

Successful output is written atomically to the two reserved root files. The
manifest uses UTF-8, LF, sorted JSON object keys, compact separators, escaped
non-ASCII text, and one trailing LF. Identical package bytes and metadata
therefore produce byte-identical manifests and checksum files in different
directories.

The checksum file is standard SHA-256 text:

```text
<64 lowercase hex>  evidence-manifest-v1.json
```

## Verify

```powershell
python .\tools\evidence\build_evidence_manifest.py verify `
  --package-root "C:\evidence\stage1-run-001" `
  --require-valid
```

Verification checks, in order:

1. the checksum file exactly matches the manifest bytes;
2. the manifest is canonical JSON and uses schema version 1;
3. the descriptor, identity, configuration, diagnostics summary, performance
   summary, and current complete file inventory rebuild the exact same bytes;
4. the package remains technically valid when `--require-valid` is supplied.

Modified, missing, extra, reordered, case-colliding, symlinked, or unsafe files
fail closed with exit code `2` and a stable `invalid evidence: <code>` message.

## UF-010 fail-closed binding

A Windows artifact is accepted only when all of the following are true:

- descriptor status is `succeeded`, `complete` is `true`, and exit code is `0`;
- `build-artifacts.txt` is sorted, unique, and exactly equals the current files
  beneath `build.rootPath`;
- the UF-010 required executable, data files, DLL, log, fingerprints, and
  artifact list are present;
- `build-fingerprints.json` is schema 1,
  `StandaloneWindows64`/`Development`, and names the canonical LocalAppData
  output identity;
- the editor and repository package-lock fingerprints agree with EH-001;
- the SHA-256 of `ShooterMover.exe` equals the EH-001 artifact checksum.

UF-010 writes `build-artifacts.txt` only after Unity exits successfully and all
required artifacts are present. Requiring that exact complete list prevents a
failed, stale, or partial Windows build from being presented as a valid
immutable artifact.

## Manifest contents

The schema at
`tools/evidence/schemas/evidence-manifest-v1.schema.json` covers:

- tool name/version and SHA-256 algorithm;
- build, content, source, package-lock, save-schema, and tuning identity;
- canonical configuration fingerprint, seed, intent fixture, quality, locale,
  viewport, diagnostics bounds, and timeouts;
- session, attempt, optional parent session, lifecycle state, validity, and
  reason codes;
- observed diagnostics bounds and summary checksum;
- performance capture state/window/sample count/profile and summary checksum;
- validated UF-010 build paths and checksums;
- sorted file inventory, byte sizes, per-file SHA-256, and inventory SHA-256.

No absolute path, machine name, username, timestamp, credential, environment
variable, or network-derived value is emitted.

## Tests

Run from the repository root:

```powershell
python -m unittest -v tools.evidence.tests.test_build_evidence_manifest
```

The suite covers the SHA-256 known vector, canonical order, byte-identical
regeneration, path normalization, modified/missing/extra files, reordered
manifest inventory, reordered UF-010 artifact inventory, invalid sessions,
failed/incomplete builds, and cross-directory reproducibility.

## Known limitations

- SHA-256 provides integrity and reproducibility, not public-key authenticity or
  signer identity.
- Opaque diagnostics and performance summary files are checksummed but not
  semantically redacted by this tool; their producers remain responsible for
  the accepted local bounded/redacted contracts.
- The two output files are individually atomic, not a filesystem transaction.
  An interruption between writes leaves an incomplete package that verification
  rejects.
- Files changing while they are being read can produce a failed verification or
  a manifest for the observed read sequence. Capture producers must close and
  freeze outputs before manifest generation.
- Case-collision checks use Unicode case folding as a machine-neutral safety
  rule and may be stricter than a particular filesystem.

## Rollback

Rollback is one unit: remove
`tools/evidence/build_evidence_manifest.py`,
`tools/evidence/schemas/evidence-manifest-v1.schema.json`,
`tools/evidence/tests/test_build_evidence_manifest.py`, and this document.
Existing evidence files remain locally readable, but without this generator and
schema they are not accepted as formally manifested Stage 1 artifacts.
