## Lifecycle boundary

The Task Splitter creates, in a separate branch and PR:

- `assembly/generated/task_batch_index.json`;
- schema-valid task batches;
- `assembly/generated/task_backlog.json`;
- updated collaboration state.

It must not silently revise requirements or architecture. Material change returns to a planning amendment.

## Decomposition sequence

Create batches in this dependency order without assigning task IDs here:

1. repository/Unity baseline, contracts, stable IDs, assemblies, generated validation;
2. Stage 1 evidence harness, arena shell, input, build identity;
3. movement/thruster domain and adapters;
4. shared combat contracts and four-mount foundation;
5. five Stage 1 weapon packages defined by `AMENDMENT_STAGE1_WEAPONS.md`;
6. four ordinary enemy roles, Four-Blaster Elite and short route as defined by `AMENDMENT_STAGE1_ENEMIES.md`;
7. Stage 1 reliability, accessibility, diagnostics, performance, formal evidence;
8. Stage 1 gate execution and explicitly approved bounded iteration;
9. Stage 2 mission state, room projection, snapshots, journal, checkpoint, banking, suspend, recovery;
10. factory graph and modular rooms;
11. remaining weapons, enemies, Prototype Overseer, complete encounters;
12. menu, shop, refresh tokens, strongboxes, reward review, loadout, records, replay;
13. full accessibility, diagnostics, build, stress, clean-machine, and save matrices;
14. representative final-art pipeline and independent reproduction;
15. Stage 2 reliability and production-readiness review.

Do not schedule Stage 2 implementation as ready work until Stage 1 passes.

## Parallel safety

- Shared contract tasks precede consumers.
- One task owns each scene, prefab, ScriptableObject, central table, generated tool, or shared module it edits.
- Content tasks stay inside owned packages plus approved generated outputs.
- Generated files are regenerated by an owner task, never manually conflict-resolved.
- Shared-system changes are separate stronger-review tasks.
- One task is one revertable change.
- Human playable review is an explicit dependency for feel, readability, pacing, audio, and UI.

## Required task-card context

Every implementation task declares milestone/frozen question, owner lane, allowed/forbidden paths, exact dependencies, consumed contract versions, acceptance criteria, automated/playable verification, proof, edge cases/non-goals, serialized ownership, save/content impact, rollback, size, cap spend, and prototype-debt entry if needed.

## Cut order

At a cap preserve evidence-critical work and cut additional polish/variants, extra content, secondary progression/economy, optional generalized tooling, then post-MVP preparation. Never cut required validation, accessibility, diagnostics, save safety, reliability, or performance merely to preserve breadth.

## Invariants

- offline guest campaign;
- fully 2D runtime;
- one authoritative engine-independent mission state;
- four independent weapons under one aim/fire intent;
- unlimited normal fire and independent power banks;
- deterministic mission rules and reward commitments;
- stable-per-run shop with mission-only refreshes;
- atomic saves, rewards, banking, and completion;
- stable IDs/generated registries;
- device-independent input;
- no post-MVP service/networking implementation;
- no mature randomized item modifiers;
- no unlicensed assets or shipped secrets;
- no implementation before dependencies and proof are explicit.
